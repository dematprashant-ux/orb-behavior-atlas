# 1. Purpose

The Backtesting Framework provides a deterministic, reproducible, and unbiased environment for evaluating trading strategies generated by the Strategy Engine.

Its objective is to estimate how a strategy would have performed under historical market conditions while minimizing common sources of simulation bias.

The framework evaluates strategies.

It never creates strategies.

## 1.1 Backtesting Engine Foundation

M10.1 establishes only the technology-neutral Backtesting Engine contract:
immutable `BacktestContext`, `BacktestRun`, and `BacktestStatus` models, pure
builders, and the `BacktestEngine` protocol. A context references existing
historical research artifacts, a strategy, and an execution-engine dependency;
a run references its context.

This foundation does not replay historical data, invoke injected services,
simulate fills, calculate PnL, returns, drawdown, Sharpe ratio, or win rate,
build equity curves, duplicate upstream engine logic, or perform I/O.

# 5. Simulation Principles

The simulation must be:

- Deterministic
- Reproducible
- Event-driven
- Chronological
- Free from look-ahead bias
- Fully auditable

Every simulation should produce identical results when executed with identical inputs.

---

# 6. Order Execution Model

Supported order types:

- Market
- Limit
- Stop
- Stop-Limit (future)

Each simulated order records:

- Order ID
- Timestamp
- Order Type
- Direction
- Quantity
- Requested Price
- Executed Price
- Execution Status

Orders execute only using information available at the simulated time.

Open

# 9. Transaction Costs

- Percentage slippage

- Volatility-adjusted slippage
- Liquidity-aware slippage
- Order-book simulation

Every completed backtest should report:

- Net Profit
- Gross Profit
- Gross Loss
- Total Trades
- Win Rate
- Loss Rate
- Average Win
- Average Loss
- Profit Factor
- Expectancy
- Maximum Drawdown
- Recovery Factor
- Sharpe Ratio
- Sortino Ratio
- Calmar Ratio
- Average Holding Time
- Maximum Consecutive Wins
- Maximum Consecutive Losses

Additional metrics may be added without breaking compatibility.

---

# 12. Bias Prevention

The framework must actively prevent:

- Look-ahead bias
- Survivorship bias
- Data leakage
- Future information usage
- Curve-fitting during simulation

Simulation results must only depend on information available at each historical timestamp.

---

# 13. Walk-Forward Integration

The framework supports walk-forward evaluation.

Typical workflow:

```
Training Period

↓

Freeze Strategy

↓

Validation Period

↓

Performance Review

↓

Next Window
```

Training data must never overlap with evaluation data.

---

# 14. Backtest Report

Each simulation produces a Backtest Report containing:

- Strategy Information
- Configuration
- Date Range
- Trade List
- Portfolio Summary
- Performance Metrics
- Drawdown Analysis
- Validation Summary
- Execution Log
- Metadata

Reports should be reproducible from stored inputs.

# 16. Error Handling

The framework should detect and report:

- Missing strategy
- Invalid historical data
- Invalid order configuration
- Portfolio inconsistencies
- Invalid timestamps
- Missing market data
- Calculation failures

Errors must include sufficient diagnostic information.

# 19. Design Principles

The Backtesting Framework must always be:

- Deterministic
- Transparent
- Reproducible
- Auditable
- Strategy-independent
- Bias-resistant

Simulation accuracy takes priority over execution speed.
